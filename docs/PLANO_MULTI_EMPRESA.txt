1. CONCEITO BASICO
================================================================================

O portal tera varias empresas cadastradas. Cada empresa devera ter a
possibilidade de escolher quantos e quais modulos quiser.

A tela de cadastro do portal deve ser feita com o nivel de acesso do perfil
administrador do sistema daquela determinada empresa que esta sendo cadastrada.
Esse usuario devera ter acesso para cadastrar tanto usuarios de niveis de
acesso inferior quanto mesmo nivel de acesso (administrador).

Atualmente, a Polimix esta cadastrada. O acesso de nivel administrador da
Polimix e "super.admin@polimix.local".

A tela de login devera ter o nome do proprio portal (nao da empresa).

================================================================================
2. REGRAS DE NEGOCIO
================================================================================

2.1 UNIDADES/FILIAIS
- O usuario administrador da empresa podera cadastrar as Unidades/Filiais
  da sua organizacao e os respectivos integrantes de cada filial.
- Quando os integrantes acessarem o portal, eles terao acesso somente as
  filiais em que estao inseridos.
- O administrador tem acesso a todas as unidades/filiais.

2.2 FILTRAGEM AUTOMATICA POR UNIDADE
- Como o usuario logado so tera acesso aos cadastros da sua respectiva
  Unidade/Filial, nao faz sentido a aplicacao do campo "Unidade" para
  selecao manual.
- Telas como "Areas" e "Centros de Trabalho" que atualmente exigem a
  selecao da Unidade devem ser ajustadas para usar a unidade da session.
- Demais telas que exigem que o usuario insira a Unidade tambem devem
  ser revisadas.

2.3 MODULOS POR EMPRESA
- Cada empresa pode escolher quais modulos quer utilizar.
- Os modulos disponiveis sao: Ativos, Ordens de Servico, Solicitacoes,
  Estoque & Compras, RAF, GEP, KPI, Plano de Manutencao, Planejamento,
  Cadastros Basicos, Pessoas/Equipes, Localizacoes.

5. TELAS QUE PRECISAM DE REVISAO (SELETOR DE UNIDADE)
================================================================================

Telas que atualmente exigem selecao manual de unidade e precisam ser
ajustadas para usar a unidade da session automaticamente:

 1. Areas (basic-registrations) - Remover seletor de unidade
 2. Centros de Trabalho (basic-registrations) - Remover seletor de unidade
 3. Criacao de Ativo (AssetCreatePanel.tsx) - Remover seletor de unidade
 4. Edicao de Ativo (AssetEditPanel.tsx) - Remover seletor de unidade
 5. Plano de Manutencao (planning/plans) - Remover seletor de unidade
 6. Programacao de OS (planning/schedules) - Remover seletor de unidade
 7. KPI (kpi/page.tsx) - Admin: manter seletor; Demais: auto-filtrar
 8. Arvore de Ativos (tree/page.tsx) - Admin: manter seletor; Demais: auto-filtrar

APIs que filtram por unitId e precisam de ajuste:
 - GET /api/basic-registrations/[entity] (scope unit)
 - POST /api/planning/plans
 - POST /api/planning/schedules
 - GET /api/tree
 - GET /api/kpi
 - POST /api/assets
 - GET /api/dashboard/corporate

6. PLANO DE IMPLEMENTACAO (6 FASES)
================================================================================

----------------------------------------------------------------------
FASE 1 - FUNDACAO MULTI-EMPRESA E MODULOS (Schema + Migracao)
----------------------------------------------------------------------

1.1 Schema do Banco (Prisma)
    - Criar tabela Module (id, name, slug, description, icon)
    - Criar tabela CompanyModule (companyId, moduleId, enabled)
    - Criar tabela UserUnit (userId, unitId) - relacao N:N
    - Adicionar campo activeUnitId ao model User
    - Popular modulos padrao

1.2 Migracao de Dados
    - Migrar Polimix como empresa existente com todos os modulos habilitados
    - Definir super.admin@polimix.local como SUPER_ADMIN
    - Criar registros UserUnit para usuarios existentes

1.3 API de Gestao de Empresas
    - CRUD /api/admin/companies (apenas super admin)
    - CRUD /api/admin/companies/[id]/modules

----------------------------------------------------------------------
FASE 2 - AUTENTICACAO E SESSION
----------------------------------------------------------------------

2.1 Login
    - Remover branding Polimix da tela de login
    - Usar nome/logo do portal
    - Remover/restringir registro publico

2.2 Session
    - Expandir SessionUser: unitId, unitIds[], companyName
    - Apos login, definir unitId como unidade ativa
    - Admin: permitir trocar entre unidades via header

2.3 Middleware
    - Validar modulos habilitados para a empresa
    - Filtrar sidebar por modulos da empresa

----------------------------------------------------------------------
FASE 3 - FILTRAGEM AUTOMATICA POR UNIDADE
----------------------------------------------------------------------

3.1 Backend (APIs)
    - Rotas unit-scoped: extrair unitId da session
    - Admin: aceitar unitId como parametro opcional
    - Nao-admin: forcar unitId da session

3.2 Frontend (Telas)
    - Remover dropdown "Unidade" das 6 telas listadas
    - Admin: manter seletor no header
    - CrudTable: unitScoped usa session automaticamente

3.3 Hooks
    - useAuth retorna unitId e unitIds
    - Criar useActiveUnit()

----------------------------------------------------------------------
FASE 4 - PERFIS DE ACESSO E PERMISSOES
----------------------------------------------------------------------

4.1 Refinar permissions.ts
    - Operacionais: view-only + editar SS/OS
    - Gestor/Planejador: acesso total
    - Super Admin: acesso total + gestao empresa

4.2 Enforce nas APIs
    - Verificar role + permissao em endpoints POST/PUT/DELETE
    - Retornar 403 para acoes nao permitidas

4.3 Enforce no Frontend
    - Esconder botoes para operacionais
    - SS e OS editaveis para operacionais

----------------------------------------------------------------------
FASE 5 - GESTAO DE USUARIOS E UNIDADES
----------------------------------------------------------------------

5.2 Tela de Usuarios

5.3 Atribuicao de Usuarios a Unidades

----------------------------------------------------------------------
FASE 6 - ADMINISTRACAO DO PORTAL (SUPER ADMIN GLOBAL)
----------------------------------------------------------------------

6.1 Dashboard do Portal

================================================================================
8. RISCOS E PONTOS DE ATENCAO
================================================================================

1. MIGRACAO DE DADOS
   - A Polimix ja tem dados. A migracao precisa preservar todos os vinculos.

2. SESSION EXISTENTE
   - Alterar SessionUser invalida sessions ativas (usuarios precisam relogar).

3. REGISTRO PUBLICO
   - Atualmente qualquer pessoa pode criar conta. Precisa ser desabilitado
     ou restrito a admin.

4. CASCATA DE UNITID
   - Formularios de ativos usam unitId para filtrar areas/centros de trabalho.
     Ao remover o seletor, a cascata precisa usar a unidade da session.

5. ADMIN MULTI-UNIDADE
   - O admin precisa de mecanismo para alternar entre unidades (seletor no header).

6. PERFORMANCE
   - Filtragem por unidade na session evita queries adicionais, mas precisa
     ser consistente em todas as APIs.

7. INTEGRACAO TOTVS
   - Exportacao de dados tem filtro por unitId que precisa ser revisado.
